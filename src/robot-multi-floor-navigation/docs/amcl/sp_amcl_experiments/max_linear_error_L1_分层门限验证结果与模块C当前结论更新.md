# max_linear_error_L1 分层门限验证结果与模块C当前结论更新

## 1. 实验目的

在前一阶段的日志分析中，已经确认当前模块C在 `large_boundary` 工况下未能稳定形成两层重定位闭环的关键原因，并不主要来自连续 K 帧判据本身，而在于 L1 阶段固定线误差门限 `max_linear_error = 0.5 m` 与实际收敛状态之间存在失配。具体表现为：在部分跑次中，系统进入 L1 后长期处于 `tr(Σ)` 很小、`C_t` 很高、但 `e_lin` 维持在约 `0.5~0.6 m` 的状态，从而导致 `frame_ok` 持续为 0，最终无法形成第二次 `RELOC_PASS`。

基于这一分析，本轮实验的主要目标不再是继续单独比较 K，而是验证如下问题：

**在仅对 L1 阶段放宽线误差门限的情况下，是否能够有效恢复第二次 `RELOC_PASS` 的形成能力，并重新打通两层重定位判定链路。**

---

## 2. 实现方式与实验设置

## 2.1 参数实现

本轮实验新增节点私参 `~max_linear_error_L1`，其逻辑为：

- 若 `max_linear_error_L1 > 0`，则在 `desired_map_level == 1` 时使用该值作为 L1 阶段线误差门限；
- 否则，L0 与 L1 均继续使用全局参数 `~max_linear_error`，默认值为 `0.5 m`。

这一实现保证了：

- L0 仍保持原有线误差判定标准；
- L1 可单独采用更宽松的门限，以适应跨层后目标层收敛误差更大的实际情况。

## 2.2 启动与脚本

- `nav.launch` 已增加同名 arg；
- `run_one_experiment.sh` 第 13 参传入；
- 一键复现实验脚本为：`scripts/amcl/run_max_linear_error_L1_verify.sh`

本轮统一采用如下实验设置：

- 扰动工况：`large_boundary`
- 扰动参数：`(0.32, 0.32, 0.2)`
- 单次时长：`240 s`
- 日志目录：`experiment_logs_max_lin_L1_verify`

本轮共进行三组单次运行：

1. baseline
2. relocc 默认配置
3. relocc 并设置 `max_linear_error_L1 = 0.65`

---

## 3. parse_experiment_logs.py 解析结果

根据 `parse_experiment_logs.py` 的解析结果，本轮三组单次样本摘要如下表所示。

| 运行 | `reloc_ok` | `task_ok` | 说明 |
|------|------------|-----------|------|
| baseline run1 | 2 | True | 对照 |
| relocc run2（默认 L1 门限） | 2 | True | 默认门限下两层通过并到达目标 |
| relocc run3（`max_linear_error_L1 = 0.65`） | 2 | False | 两层均出现 `RELOC_PASS`，但未到达最终目标 |

从表中可以看出，本轮两组 relocc 均实现了 `reloc_ok = 2`，说明在这批单次样本中，模块C均成功形成了两层重定位通过链路。但与此同时，run3 虽然完成了两层 `RELOC_PASS`，最终却未到达目标，表现为 `task_ok = False`。

这说明：

**重定位层闭环是否形成，与最终任务是否送达，并不完全等价。**

因此，后续实验分析必须将：

- 重定位层成功  
与
- 任务层成功

明确区分开来。

---

## 4. 日志与机理分析

## 4.1 L1 分层门限已经被日志直接验证为有效机制

run3 的日志中可明确观察到如下信息：

- `L1 linear error threshold: 0.650 m (L0: 0.500 m)`

这说明 L1 分层门限不仅在代码中实现，而且在运行时已经真实生效。更关键的是，日志中还记录到第二次 `RELOC_PASS` 对应的判定过程：

- `linear_error ≈ 0.49 m`
- `frame_ok = 1`
- 连续 K 帧通过
- 最终形成第二次 `RELOC_PASS`

这一链路说明，L1 分层门限的作用并非停留在“参数被传入”的层面，而是已经切实参与并影响了 L1 的通过判定。

因此，可以明确得出如下机制性结论：

**在 L1 阶段采用单独线误差门限，能够有效缓解固定 `0.5 m` 门限下长期 `frame_ok = 0` 的问题，并恢复第二次 `RELOC_PASS` 的形成能力。**

这标志着 `max_linear_error_L1` 已经从先前的“猜测性改法”，转变为“被日志直接验证过的有效机制”。

---

## 4.2 run3 的 `task_ok=False` 不宜归因于模块C“假通过”

虽然 run3 最终表现为 `task_ok = False`，但进一步检查日志可知，其主因并非模块C放宽后产生了典型“假通过”，而是后续规划与恢复层失败。具体表现为：

- 仿真时间约 `1281 s`
- `move_base` 报错：`Aborting because a valid plan could not be found`
- 多次 recovery 后仍无法生成有效路径
- 最终未出现 `Robot arrived at Goal`

这说明 run3 的实际情况是：

- 重定位层：已经完成两层通过，`reloc_ok = 2`
- 任务执行层：后续规划失败，导致未到达目标

因此，本跑次更合理的解释是：

**分层门限恢复了重定位层闭环，但并未保证规划层一定成功。**

也正因如此，`task_ok=False` 在该跑次中不宜直接归因于“模块C放松导致假通过”，而应拆解为：

1. 重定位层已经闭环成功；
2. 规划与执行层在后续阶段失败。

这一点对于论文表述非常重要。若不加区分，就容易把所有 `task_ok=False` 粗略归因于模块C本身，从而造成机理解释失真。

---

## 4.3 默认门限 run2 成功，说明问题是“鲁棒性不足”而非“绝对不可通过”

本轮默认门限的 run2 表现为：

- `reloc_ok = 2`
- `task_ok = True`

这说明在相同扰动档位、相同实验时长下，默认 `0.5 m` 门限并非绝对无法完成两层通过与任务送达。结合此前“默认门限下 L1 被 `e_lin` 卡死”的日志样本，可以更准确地说明：

**默认固定门限的问题不在于绝对不可通过，而在于边界工况下的通过能力存在明显随机性与不稳定性。**

也就是说，当前 `0.5 m` 固定门限配置在 `large_boundary` 工况下的主要问题应表述为：

- 不是始终失败；
- 而是成功不稳定、鲁棒性不足；
- 容易受 AMCL 波动、运动过程和局部状态扰动影响。

这一表述比“默认门限不可用”更准确，也更符合你目前累积得到的实验事实。

---

## 5. 当前对模块C结论的更新

基于当前这轮 `max_linear_error_L1` 分层门限验证，可以对模块C的阶段性认识更新为以下几点。

### 5.1 当前主矛盾已经从 K 转向 L1 线误差门限适配

前期 K 消融实验说明，减小 K 可以一定程度上缩短等待时间，但后续日志分析已经表明，在 L1 阶段长期 `frame_ok = 0` 的情况下，K 的变化并不会改变“单帧根本过不了”的基本事实。因此，当前模块C的主要瓶颈已不再是 K 本身，而是：

**L1 阶段固定线误差门限是否与目标层实际可达收敛误差相匹配。**

这一点已经由本轮分层门限实验得到更直接支持。

### 5.2 `max_linear_error_L1` 分层门限可以修复重定位层闭环能力

本轮 run3 已经清楚显示：

- L1 独立门限被正确调用；
- 第二次 `RELOC_PASS` 可以重新形成；
- `reloc_ok` 恢复为 2。

因此，可以将 `max_linear_error_L1` 视为：

**当前模块C在目标层判据适配上的一个有效修复方向。**

它的作用重点不是“保证任务一定成功”，而是：

**提高目标层重定位通过链路的形成能力。**

### 5.3 `reloc_ok` 与 `task_ok` 必须解耦分析

本轮 run3 已经提供了一个非常典型的样本：

- `reloc_ok = 2`
- `task_ok = False`

这说明：

- 两层重定位通过  
不等于  
- 全任务一定成功送达

因此，在后续实验统计与论文写作中，应将系统结果至少拆分为两个层面：

1. **重定位层结果**：是否形成 L0 / L1 的 `RELOC_PASS`，是否达到 `reloc_ok = 2`
2. **任务层结果**：是否到达目标，是否出现规划失败、恢复失败或路径不可解

只有这样，才能更准确地识别模块C改动到底影响的是：

- 重定位判据本身  
还是
- 系统后续的规划执行过程

## 6. 对论文写作的建议

本轮分层门限实验更适合作为：

**机制验证实验**

而不宜直接写成：

**统计显著的性能提升结论**

因为当前仍然是单次种子样本，其主要价值在于证明以下事实：

1. L1 分层门限在代码和运行时确实生效；
2. 它可以直接参与并修复第二次 `RELOC_PASS` 的形成；
3. 它打通的是“重定位层闭环能力”，而不是直接保证最终任务送达。

因此，在论文中更稳妥的表述应类似于：

> 为验证目标层固定线误差门限失配是否为阻断第二次重定位通过的关键因素，本文进一步引入 `max_linear_error_L1` 分层门限，仅在 L1 阶段放宽线误差判定阈值。单次机制验证结果表明，当 L1 阈值设为 0.65 m 时，日志中可明确观察到第二次 `RELOC_PASS` 的形成过程，说明目标层分层门限能够有效缓解固定 0.5 m 门限下长期 `frame_ok=0` 的问题，从而恢复两层重定位判定链路。需要指出的是，该结果主要说明分层门限可修复重定位层闭环能力，而不直接等价于任务最终送达成功；任务层表现仍受到后续规划与恢复行为影响。

这样的表达既能体现改动的有效性，又不会因为样本不足而过度拔高结论。

---

## 7. 下一步实验建议

基于当前结果，后续最值得优先开展的工作不再是继续大范围扫描 K，而是围绕 `max_linear_error_L1` 做小规模机制与鲁棒性验证。

### 7.1 建议先做小规模门限扫描

建议优先比较：

- `max_linear_error_L1 = 0.55`
- `max_linear_error_L1 = 0.60`
- `max_linear_error_L1 = 0.65`

重点观察：

- 第二次 `RELOC_PASS` 是否更稳定形成；
- `dt_L1` 是否明显下降；
- 是否出现明显过松通过；
- `task_ok` 是否改善；
- 规划失败是否增多。

这样可以帮助判断：

- `0.65` 是否确有必要；
- 还是 `0.60` 左右就足以改善鲁棒性；
- 分层门限放宽的合理区间大致位于哪里。

### 7.2 建议后续将失败类型分层统计

后续实验总结时，不建议继续仅用单一 `task_ok` 表示系统表现，而应补充区分：

- 重定位层失败  
- 规划层失败  
- 联合失败

例如可增加以下统计字段：

- 是否形成第二次 `RELOC_PASS`
- `reloc_ok`
- 是否出现 `valid plan could not be found`
- recovery 次数
- 未到达目标的直接原因

这样有助于在论文中更清楚地说明：

- 某次失败究竟是模块C未放行；
- 还是模块C已放行但规划层后续失效。

---

## 8. 结论

本轮 `max_linear_error_L1` 分层门限验证表明，当前模块C在 `large_boundary` 工况下的主要改进方向，已经从早期关注的 K 时序约束，转向目标层线误差门限的适配问题。实验日志显示，L1 独立门限 `0.65 m` 已经能够直接参与第二次通过判定，并有效恢复两层 `RELOC_PASS` 的形成能力。这说明 `max_linear_error_L1` 是当前修复目标层重定位闭环能力的有效机制。

与此同时，本轮 run3 也说明：

**重定位层闭环形成，并不等于任务层最终成功。**

在该跑次中，两层 `RELOC_PASS` 已全部出现，但后续仍因 `move_base` 无法生成有效路径而未能到达目标。因此，后续实验与论文分析必须将：

- 重定位判定成功  
与
- 系统任务成功

明确解耦。

综合来看，当前模块C的阶段性结论可概括为：

**L1 分层线误差门限能够有效修复目标层第二次 `RELOC_PASS` 的形成能力，但其对整体任务成功率的提升仍需通过多次重复实验，并结合规划层失败分析进一步验证。**
