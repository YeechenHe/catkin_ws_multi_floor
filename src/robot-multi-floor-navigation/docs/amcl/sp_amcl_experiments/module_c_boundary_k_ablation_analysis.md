# 模块C边界工况下 K 消融结果分析与下一步调参建议

## 1. 实验背景

本轮实验旨在以最少次数对模块C中的连续帧判据参数 `K` 进行初步消融，观察在 `large_boundary` 扰动条件下，减小 `K` 是否能够缓解当前模块C存在的长时间等待问题，并为后续正式批量实验提供趋势参考。

本次实验采用单次 pilot 设计，总计进行 3 次仿真，分别对应：

1. Baseline，run1
2. relocc，`K = 3`，run2
3. relocc，`K = 2`，run3

需要强调的是，本次实验样本量极小，仅用于趋势分析，不能作为正式统计结论依据。

---

## 2. 实验设计

### 2.1 工况设置

本次消融统一采用如下扰动条件：

- 扰动类型：`large_boundary`
- 扰动参数：`(0.32, 0.32, 0.2)`
- 单次仿真时长：`240 s`

### 2.2 实验顺序

| 顺序 | 内容 | 说明 |
|---|---|---|
| 1 | baseline，run1 | `large_boundary (0.32, 0.32, 0.2)`，240 s |
| 2 | relocc，K=3（YAML，`K_override=0`），run2 | 同扰动 |
| 3 | relocc，K=2（`K_override=2`），run3 | 同扰动 |

### 2.3 脚本与日志

- 脚本：`scripts/amcl/run_k_ablation_minimal.sh`
- 日志目录：`multi_floor_nav/experiment_logs_k_ablation_minimal/`

---

## 3. 单次实验结果

本次结果如下表所示。

| 运行 | reloc_ok（解析） | task_ok | retries | dt_L0 (s) | dt_L1 |
|---|---:|---|---:|---:|---|
| Baseline run1 | 2 | ✓ | 2 | 12.3 | 12.3 |
| relocc K=3 run2 | 1 | ✗ | 4 | 96.3 | — |
| relocc K=2 run3 | 1 | ✗ | 3 | 46.8 | — |

从结果可以看出：

- Baseline 本次成功完成两层重定位并到达目标；
- 两次 relocc 均只出现一次“到位类事件”，即 `reloc_ok = 1`；
- 两次 relocc 均未完成两层闭环流程，`task_ok` 均为失败；
- 在同一 pilot 条件下，`K = 2` 的 `dt_L0` 明显小于 `K = 3`。

---

## 4. 结果分析

## 4.1 可初步支持的趋势：减小 K 有助于缩短等待时间

本次实验中：

- `K = 3` 时，`dt_L0 ≈ 96.3 s`
- `K = 2` 时，`dt_L0 ≈ 46.8 s`

两者相比，`K = 2` 的 L0 阶段等待时间明显缩短。这说明在当前边界扰动工况下，减小连续帧要求确实有可能缓解由多帧一致性约束带来的长时间等待问题。

因此，可以将本次实验视为对以下判断的单次趋势性支持：

**减小连续帧判据参数 K，有助于降低模块C的判定耗时。**

但需要注意，这一结论目前仅停留在“等待时间缩短”的层面，并不意味着系统整体性能已经改善。

## 4.2 当前更关键的问题：模块C可能已经影响完整任务闭环

本次实验中最值得重视的现象，并不是 `K = 2` 比 `K = 3` 更快，而是：

- Baseline 成功完成两层重定位，`reloc_ok = 2`，`task_ok = ✓`
- 两次 relocc 均仅完成一次“到位类事件”，`reloc_ok = 1`
- 两次 relocc 最终均未完成任务闭环，`task_ok = ✗`

这说明当前模块C在 `large_boundary` 工况下，问题已经不仅仅表现为“更慢”，还可能表现为：

**通过判据过强，导致系统难以顺利完成完整的跨层重定位流程。**

换言之，当前模块C在边界扰动下可能已经对任务可完成性造成影响。

## 4.3 K 并不是唯一瓶颈

虽然本次实验显示，减小 `K` 能够显著缩短等待时间，但结果同时也表明：

- `K = 3` 失败
- `K = 2` 仍然失败

这说明，当前问题并不只是因为 `K` 过大。即使将连续帧要求从 3 降为 2，系统仍然未恢复完整任务成功能力。因此，后续需要进一步排查是否还存在以下因素共同导致模块C过严：

- `tau` 阈值偏严
- `reloc_min_check_delay` 过长
- 触发条件与通过条件耦合过紧
- L0 与 L1 之间的状态衔接存在阻塞
- 位姿波动或航向抖动持续打断通过判定

因此，本次实验只能说明 `K` 是一个有效的调参杆，但尚不能说明它是决定性因素。

## 4.4 当前优化目标应从“更稳”调整为“先恢复可完成性”

在之前的分析中，模块C的主要问题更多表现为：

- 判定耗时偏长
- retries 偏多
- 判据偏保守

但本次实验表明，在边界工况下，当前模块C的问题可能已经进一步表现为：

- 系统难以完成两层重定位闭环
- 判据保守性开始影响完整任务推进

因此，后续优化目标应适当调整：

**现阶段不应再优先追求更严格或更稳定，而应先恢复模块C在边界工况下的任务可完成性。**

只有在能够稳定完成两层闭环之后，才有必要进一步比较不同参数下的判定质量、throughput 与 false accept 表现。

---

## 5. 下一步建议

结合本次结果，后续工作建议按以下顺序展开。

## 5.1 优先查清失败机理，而不是直接扩大样本量

当前最重要的问题不是再多跑几次，而是先确认两次 relocc 失败的具体原因。建议优先检查日志，重点关注以下问题：

1. L0 是否真正完成并成功放行；
2. L1 是否长时间无法满足通过条件；
3. `tr(Σ)` 是始终较高，还是已经降低但连续 K 帧总是无法满足；
4. 是否存在姿态抖动或局部波动导致通过判据持续被打断；
5. retries 是在哪一阶段触发的。

只有先明确失败路径，后续调参才有针对性。

## 5.2 后续调参建议以 K=2 为基础继续推进

从本次结果看，`K = 3` 已经表现出非常明显的长等待特征，而 `K = 2` 至少在等待时间上更接近可接受范围。因此，后续更合理的做法是：

- 暂时将 `K = 2` 作为后续试探的主基础配置；
- 不建议继续以 `K = 3` 作为主要候选参数反复尝试。

但也要明确，`K = 2` 目前仍未成功，因此它只能作为“更有希望继续调”的起点，而不能视为最终解。

## 5.3 下一步应联动检查 delay、tau 与放宽机制

由于 `K = 2` 仍未恢复任务成功，因此下一轮实验不应仅继续比较不同 K，而应联动检查以下因素：

- 是否需要缩短 `reloc_min_check_delay`
- 是否需要略微放宽 `tau`
- 是否需要加入“超时后适度放宽”的机制
- 是否需要在不同阶段使用不同强度的判据

其中，建议优先检查 `reloc_min_check_delay` 与超时放宽逻辑，因为当前问题已经不只是“多等一点”，而是“可能一直等不到合理放行”。

## 5.4 下一轮实验首要目标应设为恢复两层闭环成功

后续最小实验设计不应再以“比较 K=2 和 K=3 谁更快”为首要目标，而应首先验证：

**在边界工况下，模块C是否还能恢复完整两层重定位闭环与任务成功。**

建议下一轮最小验证组合可考虑：

1. Baseline，1次
2. relocc，`K = 2`，1次
3. relocc，`K = 2` + 缩短 delay，1次
4. relocc，`K = 2` + 略放宽 tau 或加入超时放宽，1次

此时重点观察的核心指标应调整为：

- 是否恢复 `reloc_ok = 2`
- 是否恢复 `task_ok = ✓`
- 在恢复成功的前提下，`dt` 是否仍可接受

---

## 6. 结论

本次边界工况下的最少次数 K 消融实验，虽然样本量很小，但已经提供了两个重要信息。

第一，减小连续帧判据参数 `K`，确实有助于明显降低 L0 阶段的等待时间，说明 `K` 是一个有效的调参方向。

第二，更关键的是，当前模块C在 `large_boundary` 工况下的问题已经不仅是“判定偏慢”，还可能进一步表现为“影响完整任务闭环成功”。在本次 pilot 中，两次 relocc 均未恢复两层任务成功，说明当前模块C的通过逻辑仍然过强，或至少尚未与边界扰动场景形成良好匹配。

因此，后续优化工作的重点不应继续停留在单一 K 比较上，而应转向：

**以恢复边界工况下的完整任务可完成性为首要目标，联合检查 K、delay、tau 与放宽机制之间的耦合关系。**
