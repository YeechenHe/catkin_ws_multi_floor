# 模块C未形成两层闭环的日志机理分析与参数调整优先级建议

## 1. 问题背景

在 `large_boundary` 工况下开展的最少次数 K 消融实验中，Baseline 能够完成两层重定位并成功到达目标，而两次 relocc 运行均只形成一次 `RELOC_PASS`，未完成两层闭环，最终 `task_ok = false`。最初从 Summary 结果看，这一现象容易被理解为“模块C未能完成第一层通过”或“连续 K 帧判据过强”。但在进一步检查 `run2 / run3` 日志并结合代码参数后，可以确认：当前问题的主因并不在 L0，也不主要在 K，而是 L1 阶段线误差门限与实际收敛状态之间存在明显失配。

本说明文档用于总结这次日志分析得到的机理结论，并据此重新排序后续参数调整的优先级。

---

## 2. 日志时间线分析

### 2.1 L0 阶段：模块C已经成功通过

根据 `relocc_pert_0.32_0.32_0.2_run2.log` 与 `run3.log` 的日志记录，L0 阶段的关键事件如下。

#### run2（K=3）

- 在 L0 前期经历 3 次 `Setting initial pose failed`，随后进行重试并放大 sigma；
- 之后出现 `Module C: reloc passed (consecutive 3 >= K=3)`；
- 随后出现第一次 `RELOC_PASS`；
- 再进入电梯流程，并完成地图切换，日志中出现 `Successfully loaded map for level: 1`。

#### run3（K=2）

- 在 L0 前期经历 1 次重试；
- 随后出现 `Module C: reloc passed (consecutive 2 >= K=2)`；
- 出现第一次 `RELOC_PASS`；
- 后续同样进入电梯并成功切换至 L1。

由此可以明确：

**L0 阶段虽然耗时较长且伴随重试，但模块C最终是通过了第一层判定的。**

因此，Summary 中表现出的“只有一层通过”并不意味着 L0 未通过，而是说明系统在 L1 阶段未能形成第二次 `RELOC_PASS`，从而无法完成两层闭环。

---

### 2.2 L1 阶段：系统长时间卡死，最终超时重试

两份日志在进入 L1 后的模式高度一致，主要表现为：

1. 出现 `Successfully loaded map for level: 1`；
2. 随后执行 `Initializing Pose ... level: 1`；
3. 进入约 6 s 的 nomotion 等待；
4. 之后长时间输出类似信息：
   - `Linear Error: 0.60`（仅有小幅波动）
   - `tr(Σ) ≈ 0`
   - `C_t ≈ 1.0`
   - `frame_ok = 0`
5. 该状态持续到 L1 检查超时后，日志中出现：
   - `Setting initial pose failed. Will Retry`
   - `Retry #1`
   - 再次 `Initializing Pose ... level: 1`

随后整段任务在下一轮实验重启前被中断，日志末尾可见自动重启 master 的信息。整个 L1 阶段始终没有出现第二次 `RELOC_PASS`，因此最终被解析为：

- `reloc_ok = 1`
- `task_ok = false`

这与 Summary 中的结果完全一致。

---

## 3. 根因定位

## 3.1 当前主因不是 K，而是线误差门限触发失败

结合代码 `multi_floor_navigation.cpp` 中模块C相关逻辑，可以确认该模块使用参数 `~max_linear_error` 作为线误差门限，其默认值为：

- `max_linear_error = 0.5 m`

而 L1 日志显示，系统在很长一段时间内持续处于如下状态：

- `Linear Error ≈ 0.60 m`
- `tr(Σ) ≈ 0`
- `C_t ≈ 1.0`
- `frame_ok = 0`

这说明当前 L1 阶段的实际判据关系为：

- 协方差迹已经足够小，满足统计收缩性要求；
- 重定位置信度已经接近 1，满足概率意义上的收敛判断；
- 但线误差始终压不进 `0.5 m`；
- 因此每一帧都无法通过 `frame_ok` 检查；
- 连续 K 帧条件永远无法形成；
- 最终只能等到超时，然后触发 retry。

从机制上看，此时系统失败的直接原因不是“连续 K 帧太难凑齐”，而是：

**单帧条件本身就始终不成立。**

既然每一帧的 `frame_ok` 都为 0，那么无论 `K = 2` 还是 `K = 3`，最终结果都不会改变，区别只在于前期等待时间和局部触发节奏略有不同，而不会改变 L1 注定超时的结局。

因此，可以明确地说：

**当前 relocc 未形成两层闭环的主因，与 K 取值关系不大，而主要来自 L1 阶段 `max_linear_error = 0.5 m` 与实际收敛误差约 `0.60 m` 之间的不匹配。**

---

## 3.2 该现象是“tr(Σ) 小 ≠ 位姿正确”的典型实例

本次日志现象与此前对模块C局限性的理论判断高度一致。L1 阶段已经出现如下特征：

- 粒子协方差迹趋近于 0；
- 置信度指标接近 1；
- 但线误差仍稳定停留在约 `0.60 m`；
- 系统虽然“聚拢”，却没有“聚对”。

这说明：

**协方差充分收缩，并不必然意味着位姿已经达到可接受精度。**

从实验分析角度看，这一现象可以视为模块C判据局限性的一个典型反例，即：

> `tr(Σ)` 小只能说明粒子分布集中，却不能单独保证集中位置就是正确位置。

但与此同时，这次实验也暴露出另一层问题：当前线误差阈值采用固定 `0.5 m` 硬门限，在边界扰动场景下缺乏足够的适配性，以至于系统即使已经进入“接近可接受”的状态，也无法被放行，最终导致任务流程卡死。

---

## 4. retries 的来源解释

本次实验中统计得到的 `retries` 并不应简单理解为“K 太大导致的一直等”。结合日志可知，重试主要来自两个阶段：

### 4.1 L0 阶段重试

在第一次 `RELOC_PASS` 之前，系统多次出现：

- `Setting initial pose failed`
- 随后进行 retry
- 并伴随 sigma 放大

这部分重试说明模块C在 L0 前期已经存在一定收敛困难，但最终仍能完成第一层通过。

### 4.2 L1 阶段重试

进入 L1 后，系统长期处于：

- `e_lin ≈ 0.60 m`
- `frame_ok = 0`
- 连续 K 帧无法成立

最终触发 L1 检查超时，随后出现：

- `Setting initial pose failed. Will Retry`
- `Retry #1`

因此，L1 的 retry 本质上反映的是：

**在当前门限条件下，系统无法形成第二次 `RELOC_PASS`，并最终被超时逻辑判定为失败后重新尝试。**

这进一步说明，当前 `retries` 并不是单纯由时序判据 K 带来的，而是 L0/L1 两阶段失败与超时逻辑共同作用的结果。

---

## 5. 与 Baseline 的对比说明

在相同 `large_boundary` 扰动档位下，Baseline 可以完成两层重定位并成功到达目标，而 relocc 当前配置下无法完成两层闭环。这一对比至少说明以下几点：

1. 在同一扰动条件下，传统路径仍有可能使 L1 在实验时限内达到可接受状态；
2. 当前 relocc 的失败，并不是由场景本身绝对不可完成导致的；
3. 模块C当前配置在 L1 阶段确实存在门限失配问题；
4. 问题优先应从 `max_linear_error`、分层阈值和 L1 放宽机制入手，而不是继续把主要精力放在 K 的单独扫描上。

这意味着当前模块C的核心矛盾已经从“判据是否更严”转变为：

**判据中的线误差门限是否与目标层实际可达收敛精度相匹配。**

---

## 6. 参数调整优先级建议

基于当前日志分析，后续参数调整建议重新排序如下。

## 6.1 第一优先级：调整 `max_linear_error`

这是当前最直接、最明确的卡点。因为 L1 上长期出现的典型状态是：

- 实际 `e_lin ≈ 0.60 m`
- 阈值 `max_linear_error = 0.5 m`
- 结果 `frame_ok = 0`

因此，下一轮小规模机制验证应优先测试：

- `max_linear_error = 0.5`
- `max_linear_error = 0.6`
- `max_linear_error = 0.65`
- `max_linear_error = 0.7`

此处的目的不是简单放宽判据，而是验证：

**当线误差门限略微放宽后，系统是否能够恢复第二次 `RELOC_PASS`，并重新形成完整两层闭环。**

---

## 6.2 第二优先级：引入 L1 分层阈值

由于当前问题集中出现在 L1，而 L0 最终已经能够通过，因此不宜简单采用全局一刀切放宽策略。更合理的方向是：

- L0 维持相对严格的线误差门限；
- L1 使用略宽的线误差阈值。

这种分层策略更符合跨层后目标楼层重定位不确定性更大的特点，也更便于在论文中解释为：

**针对目标层初始误差更大、收敛更困难的实际情况，对重定位完成判据进行层级化设计。**

---

## 6.3 第三优先级：增加 L1 阶段放宽机制

从当前日志看，L1 已经出现一种很典型的“接近可接受但始终过不去”的状态：

- `tr(Σ)` 很小
- `C_t` 很高
- 但 `e_lin` 略高于固定阈值

在这种情况下，如果系统始终以固定 `0.5 m` 硬门限卡住，就只能不断等待直至超时。为避免这种“卡死”，建议在 L1 阶段设计适度放宽逻辑，例如：

- 前期采用严格门限；
- 若经过一定时间后，`tr(Σ)` 与 `C_t` 已长期稳定；
- 则适度放宽 `max_linear_error`；
- 或切换至降级通过逻辑。

这种机制比单纯延长 L1 超时更有意义，因为当前问题并不是“再等等可能就过”，而是“按现有门限大概率始终过不了”。

---

## 6.4 第四优先级：K 暂时降级为次要调参项

在当前阶段，不建议继续把主要实验资源投入到 K 的单独扫描上。原因在于：

- 当前 L1 的主问题是单帧 `frame_ok` 长期为 0；
- 单帧都不能通过时，连续 K 帧根本没有成立空间；
- 因此 K 不是当前主矛盾。

K 的作用主要体现在“系统已经接近可通过时，连续一致性要求是否过强”，而本次日志已经表明：L1 甚至没有进入“接近连续通过”的阶段。

因此，K 可以保留为后续二级调参对象，但在当前阶段应让位于 `max_linear_error` 和 L1 放宽逻辑。

---

## 7. 下一轮最小实验建议

基于本次日志机理分析，下一轮实验目标不应再设定为“K=2 和 K=3 谁更优”，而应首先验证：

**L1 线误差门限是否确实是阻止第二次 `RELOC_PASS` 形成的主因。**

建议采用如下最小验证组合：

1. Baseline，`large_boundary`，1 次
2. relocc，当前配置，1 次
3. relocc，适度放宽 `max_linear_error`，1 次
4. relocc，适度放宽 `max_linear_error` + L1 放宽机制，1 次

此时重点观察的不是 `dt` 本身，而是：

- 是否出现第二次 `RELOC_PASS`
- 是否恢复 `reloc_ok = 2`
- 是否恢复 `task_ok = true`
- 放宽后是否引入明显的假通过或后续不稳定

只有在 relocc 重新具备完整两层闭环能力后，才有必要进一步讨论 K、tau 和 delay 的更细致优化。

---

## 8. 结论

通过对 `relocc_pert_0.32_0.32_0.2_run2.log / run3.log` 的详细检查，可以明确得出以下结论：

1. L0 阶段虽然存在重试与较长等待，但模块C最终已经成功通过第一层判据；
2. 当前 relocc 未形成两层闭环的真正卡点在 L1，而不是 L0；
3. L1 上长期出现 `e_lin ≈ 0.60 m`、`tr(Σ) ≈ 0`、`C_t ≈ 1`、`frame_ok = 0` 的状态；
4. 这说明系统已经“聚拢”，但没有“聚对”，同时也说明固定 `max_linear_error = 0.5 m` 在边界扰动下存在适配性不足；
5. 在单帧 `frame_ok` 始终为 0 的前提下，K=2 与 K=3 的差异已不再是主要矛盾；
6. 后续优化优先应转向 `max_linear_error`、L1 分层阈值与 L1 放宽机制，而不是继续单独扫描 K。

因此，当前模块C的关键问题已经不应再概括为“连续 K 帧过严”，而应更准确地表述为：

**在边界扰动条件下，L1 阶段固定线误差门限与实际可达收敛状态不匹配，导致即使协方差充分收缩，系统仍无法形成有效通过判定，最终破坏两层重定位闭环。**

---
