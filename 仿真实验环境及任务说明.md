# 仿真实验环境及任务说明

## 1. 实验目标概述

本实验面向多楼层室内移动机器人导航问题，验证机器人在双楼层场景中完成“单层导航—电梯换层—目标楼层重定位—继续导航”这一完整跨层任务闭环的能力。系统基于 ROS 与 Gazebo 搭建，实验对象为 Clearpath Jackal 四轮差速移动机器人。

当前实验任务的核心流程为：机器人首先在 0 层完成起点到电梯口的自主导航；随后进入电梯并发送楼层切换请求；电梯将机器人运送至 1 层后，系统切换地图并重新完成定位；最后机器人在 1 层继续导航至目标点。

---

## 2. 仿真实验环境

### 2.1 仿真平台组成

实验系统基于以下平台与组件构建：

- 操作系统：Linux
- 中间件：ROS Noetic
- 三维仿真平台：Gazebo
- 移动机器人平台：Clearpath Jackal
- 定位模块：AMCL
- 路径规划与导航模块：`move_base`
- 全局规划器：`improved_astar/ImprovedAStarPlanner`
- 地图切换模块：自定义 `change_map_node`
- 跨楼层任务状态机：自定义 `multi_floor_navigation_node`

### 2.2 场景组成

实验默认加载的 Gazebo 场景文件为：

- `src/jackal_elevator/worlds/modified_elevator.world`

该场景包含以下主要元素：

- `floor0`：0 层场景
- `floor1`：1 层场景
- `elevator`：电梯轿厢模型
- `elevator_shaft`：电梯井模型
- 多个 `jersey_barrier`：用于构造室内通道、围挡和局部封闭区域
- `fire_hydrant`：用于构造具有辨识度的环境目标物

从场景结构上看，该仿真环境是一个典型的“双楼层 + 电梯 + 人工障碍物”的室内导航环境。两层楼都使用障碍物构造了较为明显的几何约束区域，并且两层的障碍布局并不完全一致，因此两层地图并非简单平移或复制关系。

### 2.3 地图配置

系统使用两张静态栅格地图分别表示两个楼层：

- `src/robot-multi-floor-navigation/multi_floor_nav/maps/map_level0.yaml`
- `src/robot-multi-floor-navigation/multi_floor_nav/maps/map_level1.yaml`

两张地图的共同参数如下：

- 地图分辨率：`0.02 m/cell`
- 地图原点：`(-10.0, -10.0, 0.0)`
- `occupied_thresh`：`0.65`
- `free_thresh`：`0.196`

运行时由两个 `map_server` 分别发布：

- `map_level_0`
- `map_level_1`

地图切换节点通过 `change_map` 服务将目标楼层地图重新发布到统一的 `map` 话题，从而为 AMCL 和导航模块提供当前楼层对应的二维环境地图。

### 2.4 Gazebo 仿真物理参数

在 `modified_elevator.world` 中，Gazebo 物理引擎采用 ODE，主要参数如下：

- `max_step_size = 0.001`
- `real_time_factor = 1.0`
- `real_time_update_rate = 1000`

该配置能够为电梯运动、机器人运动和激光传感器仿真提供较稳定的时间推进精度。

## 3. 多楼层任务说明

### 3.1 任务流程

当前系统对应的跨楼层导航任务可以概括为以下六个阶段：

1. 机器人在 0 层从起点导航至电梯前区域。
2. 机器人由电梯前进入电梯轿厢内部。
3. 系统向电梯发送楼层请求，将电梯从 0 层移动至 1 层。
4. 电梯到达 1 层后，机器人退出电梯。
5. 系统切换至 1 层地图，并在 1 层重新初始化与重定位。
6. 机器人在 1 层从电梯口继续导航至目标位置。

### 3.2 关键任务点位

根据当前仓库中的任务定义，主要任务点位如下：

| 编号 | 语义说明 | 坐标 |
|---|---|---|
| 1 | 0 层起点，靠近消防栓 | `(4.0, -5.0, 0.5)` |
| 2 | 0 层电梯前 | `(3.0, -0.5, 0.5)` |
| 3 | 0 层电梯内 | `(0.0, -0.5, 0.5)` |
| 4 | 1 层电梯内 | `(0.0, -0.5, 3.2)` |
| 5 | 1 层电梯前 | `(3.0, -0.5, 3.2)` |
| 6 | 1 层目标点，靠近消防栓 | `(4.0, 5.0, 3.2)` |

### 3.3 状态机执行逻辑

跨楼层导航由 `multi_floor_navigation_node` 的状态机控制，主流程包括：

- `LOAD_MAP`
- `INIT_POSE`
- `CHECK_INITPOSE`
- `NAV_TO_GOAL`
- `SEND_LIFT_0`
- `ENTER_LIFT_LEVEL_0`
- `SEND_LIFT_1`
- `EXIT_LIFT_LEVEL_1`
- 再次 `LOAD_MAP`
- 切换至 1 层后再次初始化与定位
- `NAV_TO_GOAL`
- `DONE`

其中，0 层导航目标点为：

- `(3.0, -0.52, π)`

1 层最终导航目标点为：

- `(4.0, 5.0, 0.0)`

系统在 0 层到达电梯前之后，不会直接进行地图切换，而是先通过电梯控制逻辑完成换层，随后在 1 层再执行地图切换和重定位过程。

## 4. 电梯插件与跨层交互机制

### 4.1 电梯模型组成

电梯模型直接定义在 Gazebo world 文件中，而不是以独立 URDF/Xacro 形式加载。其主要组成包括：

- 电梯轿厢主体
- 滑动门
- 电梯升降关节
- 电梯井结构

电梯的关键运动副包括：

1. 门关节 `door`
   - 类型：`prismatic`
   - 运动方向：沿 `y` 轴滑动
   - 上限：`1`

2. 升降关节 `lift`
   - 类型：`prismatic`
   - 运动方向：沿 `z` 轴升降
   - 上限：`10`

电梯轿厢主连杆质量为：

- `800 kg`

### 4.2 电梯插件

当前场景中与电梯相关的核心插件有两个。

#### （1）`libgazebo_ros_elevator.so`

该插件负责电梯升降与开关门控制，关键参数如下：

- `lift_joint = elevator::lift`
- `door_joint = elevator::door`
- `floor_height = 3.075`
- `door_wait_time = 10 s`
- `topic = elevator`

其中：

- `floor_height` 表示相邻楼层之间的垂直高度；
- `door_wait_time` 表示电梯门在到站后保持开启的等待时间；
- `topic = elevator` 表示电梯通过 ROS 话题接收楼层切换指令。

#### （2）`libSimEventsPlugin.so`

该插件负责对场景中预设区域进行占据事件检测，并通过事件向电梯主题发送消息。当前 world 中定义了两个区域事件：

- `region1_event`：向 `~/elevator` 发送 `0`
- `region2_event`：向 `~/elevator` 发送 `1`

该机制可用于在场景中检测机器人是否进入特定区域，并与电梯行为联动。

### 4.3 电梯调用方式

系统通过 ROS 话题 `elevator` 向电梯发送楼层请求。导航节点内部调用方式为：

- 请求前往 0 层：发送字符串 `"0"`
- 请求前往 1 层：发送字符串 `"1"`

这意味着论文中可以将电梯交互描述为“机器人导航状态机通过 ROS 话题发布楼层请求，驱动 Gazebo 电梯插件执行升降与开门动作”。

### 4.4 进出电梯策略

进入和退出电梯并不是依赖额外运动脚本独立完成，而是由导航状态机通过底盘速度控制与传感器反馈完成。

#### 进入电梯

机器人在进入电梯时采用以下策略：

- 根据前向激光扫描估计机器人正前方最近障碍距离；
- 若前方障碍过近，则立即停车，避免碰撞；
- 若前方安全，则以设定线速度缓慢前进；
- 当里程计累计位移达到设定距离后，认为机器人已进入电梯。

#### 退出电梯

机器人在电梯到达 1 层后直接执行后退运动：

- 线速度约为 `-0.25 m/s`
- 当累计退出距离达到 `3.0 m` 时停止

该实现方式体现了系统对电梯空间内狭窄运动环境的针对性处理，也反映出当前系统将“跨层切换”分解为“导航 + 电梯控制 + 位姿重建”的复合任务。

## 7. 定位与导航参数

### 7.1 初始位姿与重定位设置

为评估系统在不同初始扰动条件下的跨楼层定位与导航鲁棒性，实验中使用四类初始位姿设置：无扰动、Small 扰动、Medium 扰动和 Large 扰动。

其中，无扰动（nominal）情况下，AMCL 使用系统默认初始位姿：

- `initial_pose_x = 4.0`
- `initial_pose_y = -5.0`
- `initial_pose_a = 0.0`

在加入扰动时，实验以该默认初始位姿为基准，分别叠加不同规模的平移与偏航扰动。四类设置如下：

| 类型 | `init_pose_offset_x` | `init_pose_offset_y` | `init_pose_offset_theta` | 平面扰动规模 |
|---|---:|---:|---:|---:|
| Nominal（0 扰动） | `0.00 m` | `0.00 m` | `0.00 rad` | `0.000 m` |
| Small | `0.10 m` | `0.10 m` | `0.05 rad` | `0.141 m` |
| Medium | `0.25 m` | `0.25 m` | `0.15 rad` | `0.354 m` |
| Large | `0.40 m` | `0.40 m` | `0.20 rad` | `0.566 m` |

其中，平面扰动规模按 \(d_{planar} = \sqrt{dx^2 + dy^2}\) 计算。随着扰动规模增大，机器人初始位姿与真实位姿之间的偏差也随之增大，从而能够更系统地检验 SP-AMCL 在地图切换和跨楼层场景下的重定位恢复能力。

状态机在地图切换后会根据目标楼层重设初始化位姿：

- 0 层初始化位姿：`(4.0, -5.0, 0.0)`
- 1 层初始化位姿：`(3.0, -0.5, π)`

这使得机器人在电梯完成换层后，能够在新楼层地图中重新建立初始位姿，并继续执行后续导航任务。

## 8. 启动方式与实验入口

### 8.1 主实验入口

多楼层导航实验主入口为：

- `src/robot-multi-floor-navigation/multi_floor_nav/launch/nav.launch`

该启动文件会完成以下主要操作：

- 加载双楼层 Gazebo 场景
- 生成 Jackal 机器人模型
- 启动双地图 `map_server`
- 启动 AMCL
- 启动 `move_base`
- 启动地图切换节点 `change_map_node`
- 启动跨层状态机节点 `multi_floor_navigation_node`
- 启动 RViz 轨迹记录节点

### 8.2 机器人初始生成位置

在主导航实验中，机器人默认生成在：

- `x = 4.0`
- `y = -5.0`
- `z = 0.5`

该位置与 0 层实验起点一致。

### 8.3 建图入口

若需要重新生成楼层地图，可使用：

- `src/robot-multi-floor-navigation/multi_floor_nav/launch/mapping.launch`

该启动文件同样使用 `modified_elevator.world` 场景，并通过调整机器人高度参数适配不同楼层建图。

## 9. 可用于论文的实验环境总结

综合当前工程配置，本文仿真实验环境可以概括如下：

1. 实验基于 ROS Noetic 与 Gazebo 构建双楼层室内电梯导航场景。
2. 场景包含 0 层、1 层、电梯轿厢、电梯井以及用于构造室内布局的围挡和目标物。
3. 系统使用 Clearpath Jackal 四轮差速移动机器人作为实验平台，机器人前部搭载一套 2D 激光雷达。
4. 两层环境分别构建独立的二维静态地图，系统通过地图切换服务在跨楼层后完成地图替换。
5. 电梯通过 Gazebo 电梯插件实现升降与开关门控制，导航状态机通过 ROS 话题触发电梯动作。
6. 机器人在 0 层完成自主导航后进入电梯，换层后在 1 层重新初始化定位，并继续导航至最终目标点，从而形成完整的跨楼层导航实验闭环。
